## 2026-04-17 - 🛡️ Sentinel: Hardfork Boundary Gas Validation Asymmetry (One-Block Window)

**Vulnerability:** A critical one-block execution boundary mismatch in the `Zero6` hardfork activation logic allows a Denial of Service (DoS) on block-building. The mempool (`EthTransactionValidator`) evaluates transactions against the *current* state block's properties (where `Zero6` is inactive). However, the block builder evaluates `ArcHardforkFlags` against the *next* block's properties (where `Zero6` is active). An attacker can submit a transaction with exactly enough gas for pre-`Zero6`, slipping past the mempool, but failing deterministically during `ArcEvm::validate_initial_tx_gas` in the EVM execution step due to the new 2100/4200 SLOAD gas penalty.
**Learning:** Hardforks that introduce dynamic or static changes to transaction base/initial gas calculations *must* have synchronicity between mempool validators (which look at block `N`) and block executors (which look at block `N+1`). Asymmetric state evaluations between the TxPool and Execution layers create "poison transactions" that panic or halt block builders.
Integration testing must use end-to-end framework configurations (like `ArcTestBuilder`) to properly simulate these state transitions (e.g., crossing a hardfork block boundary in a simulated testnet) rather than isolated unit tests that manually mock handler flags.
**Prevention:** Mempools and transaction gas estimators should evaluate transaction validity by looking ahead to `latest_block_in_state.number + 1` to ensure parity with the execution layer, or use an `ArcEvmConfig` wrapper that accurately mirrors the EVM's look-ahead behavior for gas accounting.
