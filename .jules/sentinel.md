## 2026-04-17 - TxPool (Mempool) vs Block Builder Asymmetry (Denial of Service)
**Vulnerability:** A transaction that correctly passes the `validate_one_with_state` check in the `execution-txpool` could be unconditionally accepted by the mempool (e.g. `gas_limit` = 21,000 for standard transfers). However, during `execute_transaction_without_commit` inside `executor.rs`, the `ArcEvmHandler` charges an extra `PRECOMPILE_SLOAD_GAS_COST` (2,100 gas or 4,200 gas) for blocklist checks. This effectively raises the `initial_gas` to 25,200, which exceeds the `gas_limit` of 21,000. This causes `transact()` to return `CallGasCostMoreThanGasLimit`, which `execute_transaction_without_commit` wraps inside a terminal `BlockExecutionError::evm` and completely halts the block builder and network consensus (Denial of Service).
**Learning:** `execution-txpool` must precisely mirror the core execution layer (`ArcEvmHandler::validate_initial_tx_gas`) in calculating intrinsic gas. Asymmetric gas requirement leads to valid mempool transactions causing panics/fatal errors during block construction.
**Prevention:** Ensure any hardfork-specific extra intrinsic gas rules (like the extra SLOADs in Zero6) are perfectly replicated within the transaction pool validation logic so that invalid transactions are discarded before reaching the block builder.
