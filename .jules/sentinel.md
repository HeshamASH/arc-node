## 2026-04-18 - [Critical] Consensus Compliance Gap: Missing EIP-4895 and EIP-4788 Execution in ArcBlockExecutor
**Vulnerability:** The Arc Network's `ArcBlockExecutor` entirely drops EIP-4895 (Validator Withdrawals) and EIP-4788 (Beacon Block Root system calls). Withdrawals do not process (locking staking funds), and cross-chain tracking via the beacon root contract is non-functional.
**Learning:** Customizing a core `BlockExecutor` (via `alloy-evm` overriding) requires extreme diligence. Bypassing the default `apply_post_execution_changes()` meant irregular state transition mechanisms (which live outside standard EVM CALL opcode semantics) were "deleted" from the system entirely.
**Prevention:** When forking or overriding consensus-critical execution components, all lifecycle hooks (`apply_pre_execution_changes`, `apply_post_execution_changes`) should explicitly trace and replicate the upstream node reference (e.g. standard `reth`). Comprehensive E2E tests covering every EIP standard active on the network must act as regression checks to enforce feature parity.

## 2026-04-18 - [Critical] Fork Choice Deadlock and Total Supply Pseudo-Inflation
**Vulnerability:** The Arc Network's Malachite consensus integration has a critical error handling gap. When `ArcBlockExecutor` rejects a BFT-decided block as Invalid (e.g. invalid state root), the `set_latest_forkchoice_state` API call returns an Error. This `Decision::Failure` forces Malachite to endlessly call `restart_height` for the same block. This permanently deadlocks the node. Furthermore, `reward_beneficiary` leaks fees directly to validators bypassing the `NativeCoinAuthority` ledger, breaking the 1:1 fiat-collateralized total supply invariants (pseudo-inflation).
**Learning:** BFT consensus and Execution layers must agree on failure states. If the execution layer definitively rejects a payload, the consensus layer cannot simply "restart" the height expecting a different result from an immutable quorum decision.
**Prevention:** Implement a hard-halt or specific slashing/rollback protocol when BFT and EVM fundamentally disagree on block validity. For tokenomics, all minting, burning, and fee transfers for a fiat-backed Layer 1 *must* be strictly coordinated through the central Authority precompile to ensure total supply invariants are never broken by `journal_mut().balance_incr()`.

## 2026-04-18 - [High] EIP-4844 Incompatibility (Blob Kill-Switch)
**Vulnerability:** The Arc Network completely disables EIP-4844 Blob Transactions. The `ArcBlockAssembler` passes `None` for `blob_gas_price` during transaction selection, preventing blob transactions from leaving the mempool. Furthermore, it hardcodes `.with_sidecars(BlobSidecars::Empty)` on the payload, deliberately discarding blob data.
**Learning:** Hard-fork overrides (like Cancun/Prague) in custom nodes can selectively disable crucial ecosystem scaling features. While it might simplify node operations temporarily, it critically breaks compatibility for any Layer 2 ecosystem depending on that data availability.
**Prevention:** When introducing custom payload builders, all EIP features of the active hardfork must be explicitly preserved or clearly documented as a protocol deviation. Test suites must include e2e coverage asserting that L2 data layers (Type 3 transactions) successfully propagate and settle.

## 2026-04-18 - [Critical] Systemic State Divergence via Malachite/ArcBlockExecutor Deadlock
**Vulnerability:** A malicious proposer can equivocate, proposing both a valid BFT/EVM block and an EVM-invalid (poison pill) block. If the network reaches consensus on the poison pill block, `ArcBlockExecutor` will correctly reject it, returning an `Invalid` payload status. This triggers `Decision::Failure` in `decided.rs`, leading to `restart_height`. Since BFT has already committed, the node permanently loops in a Fork Choice Deadlock.
**Learning:** Decoupling consensus (Malachite) and execution (EVM) layers introduces critical synchronization boundaries. If the BFT engine assumes a block can be "restarted" after finalization, but the EVM definitively rejects it as mathematically invalid, the network halts.
**Prevention:** Ensure BFT consensus validators thoroughly execute and validate the payload *before* signing Prevotes or Precommits. If a node receives a `CommitCertificate` for an execution-invalid block, it must hard-halt or trigger a slash/rollback mechanism, rather than blindly looping `restart_height`.
